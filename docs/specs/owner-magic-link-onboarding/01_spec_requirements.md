# Owner Magic-Link Onboarding — Specification

> Sub-feature of `company-provisioning`. Goal: when a store (company) is
> provisioned, the owner's invite email must land them on the
> **set-password page** so they choose their own password on first entry —
> no temporary password, no full self-signup.

## 1. Context

Company provisioning (`POST /api/admin/companies`, backend Go) already:

- creates the owner **auth user** with `email_confirm: false`
  (`docs/specs/company-provisioning/04_provisioning_api_pseudocode.md`, step 2),
- creates `accounts` + `profiles(role=owner)`,
- dispatches an onboarding link via
  `admin.auth.admin.generateLink({ type: "invite", email })` (step 5).

The frontend already ships the whole landing chain:

- `src/app/auth/callback/page.tsx` — turns an email link (implicit
  `#access_token=` **or** PKCE `?code=`) into a session, then
  `router.replace(safeNext(next))`. `safeNext` only allows same-origin
  relative paths.
- `src/app/(auth)/reset-password/page.tsx` — confirms the session and calls
  `supabase.auth.updateUser({ password })`; renders an "invalid/expired link"
  state when no session exists.

**The single missing piece:** `generateLink` is called without a
`redirectTo`, so Supabase redirects the verified invite to the default
**Site URL** (lands on `/` or `/dashboard`), *not* on `/reset-password`.
The owner ends up signed in with a random/empty password and never sets one.

## 2. Objective

Make the invite link resolve to the set-password page by having the backend
pass `redirectTo = <APP_BASE_URL>/auth/callback?next=/reset-password`.
No frontend code change is required; frontend behavior is pinned by tests.

## 3. Functional Requirements

- **FR-1** The provisioning handler MUST call `generateLink` with
  `options.redirectTo` set to
  `<APP_BASE_URL>/auth/callback?next=%2Freset-password`.
- **FR-2** `<APP_BASE_URL>` MUST come from configuration (an existing env var,
  e.g. the public app URL already used elsewhere in the backend) — never a
  hard-coded literal.
- **FR-3** The same `redirectTo` MUST be applied to the **resend-invite** path
  (`POST /api/admin/companies/{id}/resend-invite`) so re-sent links behave
  identically to the first invite.
- **FR-4** On clicking the invite link, an authenticated recovery/verified
  session MUST exist by the time `/reset-password` loads, so the owner can
  submit a new password with `updateUser`.
- **FR-5** After the owner sets a password successfully, the existing
  `/reset-password` success UI applies (link to `/login`); no additional
  redirect logic is introduced.

## 4. Non-Functional Requirements

- **NFR-1 (no new secrets):** no password material is generated, stored, or
  emailed. The invite carries a single-use, short-lived Supabase token only.
- **NFR-2 (config, not code):** the redirect target is derived from env; the
  Supabase **redirect allow-list** MUST include `<APP_BASE_URL>/auth/callback`
  (already tracked as pending in `company-provisioning/TASKS.md:59-62`).
- **NFR-3 (idempotent frontend):** no change to `/auth/callback` or
  `/reset-password`; their behavior is locked by regression tests (Phase 03).
- **NFR-4 (fail-open provisioning):** link dispatch stays fire-and-forget —
  a mail/link failure MUST NOT roll back the created company (owner can
  recover via password-reset). Matches existing step-5 contract.

## 5. Edge Cases

- **EC-1 Expired / cold-opened link:** `/reset-password` already renders the
  "invalid link" state (`invalidLinkTitle/Body`) with a route to
  `/forgot-password`. No new handling needed; covered by a test anchor.
- **EC-2 `next` tampering / open-redirect attempt:** `safeNext` rejects
  absolute or protocol-relative (`//`) values and falls back to `/dashboard`.
  A crafted `next=https://evil.tld` MUST NOT redirect off-origin.
- **EC-3 Owner already has a password (re-invite after first login):**
  re-sending still lands on `/reset-password`; submitting simply rotates the
  password. Acceptable and non-destructive.
- **EC-4 `APP_BASE_URL` misconfigured (missing/trailing slash):** builder MUST
  trim a trailing slash before appending `/auth/callback` (mirror
  `inviteUrl()` slash hygiene in `src/lib/auth/invitations.ts:70`).
- **EC-5 Implicit vs PKCE transport:** invite links generated by admin use the
  implicit `#access_token=` transport; `/auth/callback` already handles both.
  No branch added.

## 6. Invariants

- **INV-1** Frontend files `auth/callback/page.tsx` and
  `reset-password/page.tsx` remain **unchanged** by this feature.
- **INV-2** The `next` value is always the literal same-origin path
  `/reset-password` (URL-encoded when embedded); never an absolute URL.
- **INV-3** No temporary/plaintext password ever exists in the system.

## 7. Out of Scope

- Member invites (`admin`/`agent`/`viewer`) via `/join/<token>` — unchanged.
- Supabase email-template copy and SMTP setup (ops config, not code).
- Any first-login `must_change_password` gate (explicitly rejected in favor
  of magic link).
