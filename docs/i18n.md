# Internationalization (i18n)

The app is translated with [`next-intl`](https://next-intl.dev). The
visitor's language is **auto-detected from the browser** (`Accept-Language`)
— there is no manual switcher and no `[locale]` segment in the URL.

Supported locales: **`en`** (default), **`pt`** (pt-BR), **`es`**.

This doc is both a reference for how it's wired and a **handoff** for the
in-progress work of translating every screen. See
[Migration status](#migration-status) for what's done and what's left.

---

## How it works

### Locale detection (server-side, per request)

`src/i18n/locale.ts` → `resolveLocale()` negotiates the best match between
the browser's `Accept-Language` header and the locales we ship
(`@formatjs/intl-localematcher` + `negotiator`), falling back to `en`.

It reads `headers()`, which makes routes dynamic — fine here, the whole app
is already dynamic (per-user auth cookies).

### Wiring

- `src/i18n/config.ts` — single source of truth: `locales`, `defaultLocale`,
  `localeNames`, `isLocale()`.
- `src/i18n/request.ts` — next-intl's `getRequestConfig`: resolves the locale
  and loads its catalog. Wired via `createNextIntlPlugin('./src/i18n/request.ts')`
  in `next.config.ts`.
- `src/app/layout.tsx` — sets `<html lang={locale}>` and wraps the tree in
  `<NextIntlClientProvider>` so **client** components can translate too.
- `src/messages/{en,pt,es}.json` — the message catalogs.

### Why no URL-based locale

Every route is behind auth and `robots: index:false`, so per-locale URLs buy
nothing and would force every route under `app/[locale]/` plus a locale-aware
proxy. Cookie/URL-free detection was the deliberate, lower-risk choice.

## How to translate a component

1. **Client component** (`'use client'`):

   ```tsx
   import { useTranslations } from 'next-intl';

   function MyPanel() {
     const t = useTranslations('settings.security'); // namespace
     return <h2>{t('title')}</h2>;
   }
   ```

2. **Server component**:

   ```tsx
   import { getTranslations } from 'next-intl/server';

   export default async function Page() {
     const t = await getTranslations('dashboard');
     return <h1>{t('title')}</h1>;
   }
   ```

3. Add the key to **all three** catalogs (`en`, `pt`, `es`). Key parity is
   enforced — see [Verifying](#verifying-a-batch).

### Patterns to reuse

- **Interpolation / rich text** — for a string with an inline element
  (e.g. a highlighted email), use `t.rich` with an ICU tag:

  ```tsx
  {t.rich('checkEmailBody', {
    email,
    highlight: (chunks) => <span className="text-foreground">{chunks}</span>,
  })}
  ```
  ```json
  "checkEmailBody": "We've sent a link to <highlight>{email}</highlight>."
  ```

- **Pluralization** — ICU plural, not string concatenation:

  ```tsx
  {t('unreadConversations', { count: totalUnread })}
  ```
  ```json
  "unreadConversations": "{count, plural, one {# unread conversation} other {# unread conversations}}"
  ```

- **Labels defined in module-level `const`s** (arrays/maps outside the
  component, e.g. nav items, section metadata) can't call the hook. Store a
  **translation key** on the object instead of the literal, and translate at
  render: `t(item.labelKey)`. See `src/components/layout/sidebar.tsx`
  (`navItems`) and `src/components/settings/settings-rail.tsx` (translates
  `SECTION_META` by section `id`).

### What NOT to translate

- Brand/product name (`CRM Template for WhatsApp`).
- Error messages returned by **Supabase / Meta** (they arrive already
  localized-or-not from the provider; not ours to key).
- Example values that are format, not language (`you@example.com`).

## Catalog conventions

- One top-level namespace per feature area: `common`, `auth`, `nav`, `roles`,
  `settings`, … Nest sub-sections (`settings.security`, `settings.sections`).
- `common` holds cross-cutting words (Save, Cancel, Delete, …). Reuse it
  before adding a duplicate.
- `roles` (owner/admin/agent/viewer) is shared — reuse it wherever a role
  label is shown (sidebar chip, members roster, …).

## Verifying a batch

Run after each area, before committing:

```bash
# 1. Key parity across the three catalogs (must all match)
node -e '
const fs=require("fs");
const load=l=>JSON.parse(fs.readFileSync(`src/messages/${l}.json`,"utf8"));
const flat=(o,p="")=>Object.entries(o).flatMap(([k,v])=>typeof v==="object"&&v?flat(v,p+k+"."):[p+k]);
const en=flat(load("en")).sort(), pt=flat(load("pt")).sort(), es=flat(load("es")).sort();
const diff=(a,b,an,bn)=>a.filter(k=>!b.includes(k)).map(k=>`${an}\\${bn}: ${k}`);
const p=[...diff(en,pt,"en","pt"),...diff(pt,en,"pt","en"),...diff(en,es,"en","es"),...diff(es,en,"es","en")];
console.log("keys:",en.length,pt.length,es.length, p.length?"\n"+p.join("\n"):"| parity OK");
'

# 2. Types + 3. Build
npx tsc --noEmit -p tsconfig.json
npm run build
```

## Adding a new language

1. Add the code to `locales` in `src/i18n/config.ts` and a label in
   `localeNames`.
2. Copy `src/messages/en.json` to `src/messages/<code>.json` and translate
   every value (keys stay identical — parity is enforced).
3. That's it — detection and loading are automatic.
